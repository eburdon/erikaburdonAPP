Distributed Facebook Analysis: Implementing a Microservice Architecture

Any user can download their personal Facebook data from the settings page. Using this data, this project is a distributed, open-source tool that can analyze a user's Facebook habits based on the messages they've sent to friends.

The first iteration of this project implemented MapReduce to analyze message history. Written in Go, this distributes and parallelizes processing of their message data since the day they joined Facebook. Data is stored in-memory using an efficient Prefix Trie. The final output is a pie chart of their most active hours.

The second iteration is a proof of concept that implements a server-based microservice architecture. Users upload their file via this web interface, which triggers pre-processing on an EC2 instance, followed by MapReduce on an EMR cluster, and finally post-processing on the same EC2 instance to generate a png graph. This graph is displayed in-context on the upload page.

The third iteration hopes to implement a serverless microservice using Amazon Lambda to trigger the EC2 processing and EMR analysis independently.

There's lots of future work to be done, including: a) more fine-grained analysis such as most active day of the week, month, per year, etc..., b) exploring other data provided in the data set, c) adding fault tolerance.

Frequent Itemset Mining at Low support levels

Continued work on data mining and software engineering research already done in this field. Tried to improve performance for a new algorithm, a variant of the existing Apriori algorithm standard for calculating frequently co-occurring items in transactions. The new algorithm is an improvement by requiring a 1% support level instead of 5%. This change returns more useful items in a reccommendation system.

I tried to improve phase input partitioning by implementing a custom CombineInputFormat, RecordReader, RecordWriter, and OutputFormat. If successful, the solution would merge files generated from each map cluster (potentially exponentially larger than others) into one, then resplit evenly onto the next map cluster. This would evenly distribute the workload between servers and improve performance while maintaining a total order.

Data set: Retail.dat. Written in: Java. Supporting Framework: Apache Hadoop
